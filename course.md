# Архитектура проекта

## 1) Из каких модулей состоит проект и как они связаны

Проект устроен как классическое веб-приложение из 3 логических частей:

- **Backend (`server.js`)**  
  Сервер на `Express` поднимает API и раздает статические файлы из папки `public`.
- **База данных (`data/tourism.db`)**  
  Хранит страны и контент для их страниц: заголовки, описания, пути к фото, список мест и советы.
- **Frontend (`public/*`)**  
  Обычные HTML/CSS/JS страницы:
  - `index.html` + `app.js` — главная страница и список стран;
  - `country.html` + `country.js` — детальная страница страны;
  - `styles.css`, `country.css` — стили.

Как это связано:

1. Браузер открывает страницу (например, `index.html`).
2. JS на странице делает запросы в API (`/api/countries`, `/api/countries/:id`).
3. Сервер читает данные из SQLite и возвращает JSON.
4. Frontend рендерит карточки и контент на основе ответа API.

---

## 2) Какой архитектурный паттерн используется и почему

Здесь используется **монолитный server-rendered/static + API подход** (по сути простой **клиент-серверный монолит**, близкий к MVC по разделению ролей, но без полноценного слоя шаблонов и контроллеров по папкам).

Почему такая структура выбрана удачно для этого проекта:

- проект небольшой, нет смысла усложнять React/Next или микросервисы;
- нужен минимальный backend и SQLite — это быстро поднимается в одном процессе;
- статические страницы + легкий API дают простой деплой и легкую поддержку;
- для учебного/портфолио проекта такой формат самый практичный: минимум инфраструктуры, максимум понятности.

То есть это не SPA и не микросервисы, а **один Node-процесс**, который одновременно:
- обслуживает API,
- отдает фронтенд,
- работает с БД.

## 3) Точка входа и путь от запуска до полной загрузки

### Точка входа

Точка входа приложения — файл **`server.js`**.  
Он запускается командами:

- `npm run dev`
- или `npm start`

(обе команды в `package.json` вызывают `node server.js`).

### Пошагово, что происходит

1. **Node запускает `server.js`**.
2. Создается `Express`-приложение, подключаются middleware:
   - `express.json()` для JSON-запросов,
   - `express.static("public")` для раздачи фронтенда и ассетов.
3. Вызывается `initializeDatabase()`:
   - создается таблица `countries`, если ее нет;
   - при необходимости добавляются нужные колонки;
   - если БД пустая, выполняется сидинг стартовых стран.
4. Регистрируются роуты API:
   - `GET /api/countries`
   - `GET /api/countries/:id`
   - `POST /api/countries`
   - а также роут страницы `GET /country/:id`.
5. После успешной инициализации БД сервер начинает слушать порт (`3000` по умолчанию).
6. Пользователь открывает `http://localhost:3000`:
   - сервер отдает `public/index.html`.
7. В браузере запускается `public/app.js`:
   - делает `fetch('/api/countries')`,
   - отрисовывает карточки стран.
8. При переходе в страну (`/country.html?id=2` или `/country/2`):
   - загружается `country.html`,
   - `country.js` делает `fetch('/api/countries/:id')`,
   - рендерит контент: описание, карточки мест, советы.

Итог: загрузка начинается в `server.js`, а визуальная сборка страниц завершается в браузере после запросов к API.

## 4) Самый важный пользовательский сценарий и путь данных

Самый важный сценарий в текущем приложении:  
**пользователь открывает главную, кликает по карточке страны и смотрит детальную страницу**.

Пошаговый путь данных:

1. Пользователь открывает главную (`/`).
2. `app.js` делает запрос `GET /api/countries`.
3. Сервер в `server.js` читает таблицу `countries` из SQLite и отдает JSON со списком стран.
4. На основе этого JSON фронтенд рисует карточки и ссылки вида `country.html?id=<id>`.
5. Пользователь кликает на карточку (например, Куба).
6. Открывается `country.html?id=2`.
7. `country.js` читает `id` из URL и делает `GET /api/countries/2`.
8. Сервер снова читает SQLite (одна запись по `id`) и возвращает детальные данные: описание, места, советы, пути к изображениям.
9. `country.js` подставляет эти данные в DOM (заголовок, текст, карточки мест, список советов).

### Где в этом пути есть сохранение в БД

В пользовательском режиме просмотра сохранения **нет** (только чтение).  
Сохранение реализовано в API:

- `POST /api/countries` — добавляет новую страну в SQLite.

Если нужен сценарий «клик -> сохранение», его можно сделать через форму «Добавить страну», которая будет отправлять `POST` запрос с JSON.

## 5) Управление состоянием (State Management)

В проекте используется **простое локальное состояние без библиотек** (без Redux/Vuex и т.д.).

Где хранятся данные во время работы:

- **Источник истины**: SQLite (`data/tourism.db`).
- **На сервере**: данные живут кратко в памяти во время обработки конкретного запроса.
- **На клиенте**: данные хранятся в переменных JS (`countries`, `country`) внутри `app.js` и `country.js`, затем отображаются в DOM.

То есть состояние в этом проекте:
- персистентное (долговременное) — в БД,
- временное (на время рендера) — в памяти браузера и сервера.

## 6) Как клиент общается с сервером (формат запросов/ответов)

Клиент использует стандартный `fetch()` и общается с сервером по HTTP.

### Основные эндпоинты

- `GET /api/countries`
- `GET /api/countries/:id`
- `POST /api/countries`

### Формат

- **Запросы**:
  - `GET` без тела;
  - `POST` с JSON телом и заголовком `Content-Type: application/json`.
- **Ответы**:
  - JSON (`application/json`);
  - при успехе — данные страны/списка;
  - при ошибке — JSON с полем `message`.

### Пример ответа `GET /api/countries/:id`

```json
{
  "id": 2,
  "name": "Куба",
  "subtitle": "Гавана, ритмы, ретро",
  "description": "....",
  "cardGradient": "linear-gradient(...)",
  "heroImage": "/images/countries/cuba/hero.jpg",
  "heroTitle": "Куба: остров свободы и ритма",
  "whatToSee": [
    { "title": "Гавана", "text": "...", "image": "/images/countries/cuba/havana.jpg" }
  ],
  "tips": ["...", "..."]
}
```

Итог: клиент и сервер общаются простым REST-подобным JSON API, что хорошо подходит для маленького учебного fullstack-проекта.

## 7) Структура базы данных

В проекте используется SQLite-база `data/tourism.db`.

Сейчас есть **одна основная таблица**:

- `countries`

Поля таблицы `countries`:

- `id` — `INTEGER PRIMARY KEY AUTOINCREMENT`
- `name` — название страны (`TEXT NOT NULL`)
- `subtitle` — короткий подзаголовок (`TEXT NOT NULL`)
- `description` — описание страны (`TEXT NOT NULL`)
- `card_gradient` — градиент карточки на главной (`TEXT NOT NULL`)
- `hero_image` — путь/URL hero-изображения (`TEXT`)
- `hero_title` — заголовок на странице страны (`TEXT`)
- `what_to_see` — JSON-строка с массивом мест (`TEXT`)
- `tips` — JSON-строка с массивом советов (`TEXT`)
- `created_at` — дата создания (`DATETIME DEFAULT CURRENT_TIMESTAMP`)

### Связи и внешние ключи

Отдельных связанных таблиц сейчас нет, поэтому:

- внешние ключи (`FOREIGN KEY`) отсутствуют,
- связи между таблицами не используются (одна таблица хранит все данные о стране).

## 8) Важное обращение к БД и как оно работает

Одно из самых важных обращений — получение полной информации о конкретной стране (`GET /api/countries/:id`), потому что именно оно заполняет всю страницу страны.

Фрагмент из `server.js`:

```js
const row = await get(
  db,
  `SELECT id, name, subtitle, description, card_gradient AS cardGradient, hero_image AS heroImage,
          COALESCE(hero_title, name) AS heroTitle, COALESCE(what_to_see, '[]') AS whatToSee,
          COALESCE(tips, '[]') AS tips
   FROM countries
   WHERE id = ?`,
  [id]
);
```

Как работает этот запрос:

1. Ищет ровно одну запись страны по `id`.
2. Переименовывает поля SQL в имена, удобные для фронтенда (`cardGradient`, `heroImage`, `heroTitle`).
3. Через `COALESCE` подставляет безопасные значения по умолчанию:
   - если `hero_title` пустой, берется `name`;
   - если `what_to_see`/`tips` пустые, отдается `'[]'`.
4. После запроса сервер делает `JSON.parse(...)` для `whatToSee` и `tips`, чтобы фронтенд получил уже массивы, а не строку.

Это ключевая точка, потому что здесь формируется весь payload для детальной страницы страны.

## 9) Безопасность данных в проекте

### Что уже реализовано

- **Защита от SQL-инъекций**:  
  используется параметризованный SQL с плейсхолдерами `?` и передачей параметров массивом (`[id]`, `[name, ...]`), а не склейка SQL-строк.
- **Базовая валидация ввода** (`POST /api/countries`):  
  проверка обязательных полей (`name`, `subtitle`, `description`, `cardGradient`).
- **Базовая XSS-защита на клиенте**:  
  в `country.js` есть `escapeHtml(...)` перед вставкой пользовательских строк в HTML карточек.

### Что пока не реализовано (и это нормально для учебной минимальной версии)

- аутентификация/авторизация;
- хэширование паролей (паролей в проекте нет);
- CSRF-защита;
- строгая серверная санитизация текстовых полей (например, через whitelist/validator);
- ограничение размера и формата входящих данных.

### Что можно добавить при развитии проекта

- `helmet` для HTTP security headers,
- `express-rate-limit` для защиты API,
- `zod`/`joi` для строгой схемной валидации входного JSON,
- хранение `what_to_see` и `tips` в отдельных таблицах для более строгой структуры данных.

## 10) Три самых важных файла с основной логикой

Ниже 3 файла, где сосредоточено почти все поведение приложения:

1. `server.js` (backend + API + БД)
2. `public/app.js` (логика главной страницы)
3. `public/country.js` (логика детальной страницы страны)

### 10.1 `server.js` — разбор по блокам

**Импорты и инициализация**
- Подключаются `path`, `express`, `sqlite3`.
- Создается приложение `app`.
- Определяются `PORT` и путь к базе `DB_PATH`.
- Создается папка `data`, чтобы SQLite-файл можно было создать без ошибки.

**Middleware**
- `app.use(express.json())` — сервер умеет читать JSON-тело запроса.
- `app.use(express.static(...))` — раздача фронтенда (`public`).

**Вспомогательные функции БД**
- `createDb()` — открывает соединение с SQLite.
- `run(db, sql, params)` — обертка над `db.run` в `Promise` (для INSERT/UPDATE/DDL).
- `all(db, sql, params)` — обертка над `db.all` в `Promise` (получить массив строк).
- `get(db, sql, params)` — обертка над `db.get` в `Promise` (получить 1 строку).

**`initializeDatabase()`**
- Создает таблицу `countries`, если ее нет.
- Проверяет колонки через `PRAGMA table_info`.
- При необходимости добавляет недостающие колонки (`ALTER TABLE ... ADD COLUMN ...`).
- Проверяет, пуста ли таблица.
- Если пуста — заполняет стартовыми странами (seed).
- Закрывает БД.

**Роут `GET /api/countries`**
- Открывает БД.
- Делает `SELECT` списка стран для карточек главной.
- Возвращает JSON-массив.
- При ошибке возвращает 500.
- В `finally` закрывает БД.

**Роут `GET /api/countries/:id`**
- Берет `id` из URL.
- Делает `SELECT` одной страны.
- Использует `COALESCE`, чтобы подставлять значения по умолчанию.
- Преобразует `what_to_see` и `tips` из JSON-строк в массивы.
- Отдает готовый JSON для страницы страны.

**Роут `POST /api/countries`**
- Читает поля из `req.body`.
- Делает базовую валидацию обязательных полей.
- Записывает новую страну в БД параметризованным `INSERT`.
- Возвращает созданный объект.

**Роут `GET /country/:id`**
- Отдает статический `country.html` для красивого URL.

**Запуск приложения**
- Сначала вызывается `initializeDatabase()`.
- Только после успеха запускается `app.listen(...)`.
- Если инициализация не удалась — лог ошибки и `process.exit(1)`.

---

### 10.2 `public/app.js` — главная страница

**`loadCountries()`**
- Находит контейнер карточек (`#countries-grid`).
- Запрашивает `GET /api/countries`.
- Проверяет `response.ok`.
- Парсит JSON.
- Очищает контейнер.
- Для каждой страны:
  - создает элемент карточки;
  - ставит фон (hero image или gradient);
  - вставляет название, subtitle и ссылку в страну;
  - добавляет карточку в DOM.
- В `catch` показывает fallback-текст об ошибке загрузки.

**`loadCountries();`**
- Немедленно запускает рендер при загрузке скрипта.

---

### 10.3 `public/country.js` — страница страны

**`getCountryIdFromUrl()`**
- Сначала ищет `id` в query-параметре (`?id=...`).
- Если нет — пытается взять `id` из path (`/country/:id`).

**`escapeHtml()`**
- Экранирует спецсимволы (`&`, `<`, `>`, `"`, `'`) перед вставкой в HTML.

**`loadCountry()`**
- Берет `id`.
- Находит DOM-элементы (заголовок, описание, контейнер мест, список советов).
- Делает `GET /api/countries/:id`.
- При успехе:
  - обновляет `document.title`;
  - вставляет заголовок и описание;
  - для каждого места создает карточку с картинкой и текстом;
  - для каждого совета создает `<li>`.
- Если массивов нет — показывает fallback-сообщения.
- При ошибке (404/500) показывает пользовательский fallback.

**`loadCountry();`**
- Запускает загрузку данных сразу при открытии страницы.

## 11) Сложные или неочевидные решения в коде

В проекте нет «тяжелых алгоритмов» (сортировок, графов, динамики и т.д.), но есть несколько важных неочевидных решений, которые можно уверенно рассказать на защите:

1. **Promise-обертки над SQLite callback API**  
   Вместо callback-hell (`db.run(..., cb)`), код использует `run/all/get` с `Promise`.  
   Это делает backend линейным через `async/await`, проще читать и поддерживать.

2. **Ленивая миграция схемы через `PRAGMA table_info` + `ALTER TABLE`**  
   При старте приложение проверяет, есть ли нужные колонки, и добавляет недостающие.  
   Это позволяет безопасно обновлять структуру БД без ручной пересборки базы.

3. **Гибрид маршрутов страны (`/country/:id` и `country.html?id=...`)**  
   В клиенте поддержаны оба формата `id`, чтобы страница открывалась в разных условиях раздачи/деплоя.

4. **Хранение вложенных данных в JSON-строках (`what_to_see`, `tips`)**  
   Это упрощает схему SQLite (одна таблица), но требует `JSON.stringify` при записи и `JSON.parse` при чтении.

5. **Параметризованные SQL-запросы (`WHERE id = ?`)**  
   Защищают от SQL-инъекций и отделяют SQL от пользовательских данных.

Если на защите спросят «где самая сложная логика», лучший ответ:
- `initializeDatabase()` (создание таблицы + миграции + seed),
- `GET /api/countries/:id` (формирование payload для детальной страницы),
- `country.js` (рендер сложной структуры DOM из JSON API).

## 12) Подготовка к защите: каверзные вопросы и ответы

### 12.1 Пять сложных вопросов по архитектуре и коду (и как на них отвечать)

#### Вопрос 1
**Почему `what_to_see` и `tips` хранятся JSON-строками в одной таблице, а не в отдельных нормализованных таблицах?**

**Ответ:**  
Это осознанный компромисс под масштаб учебного проекта: мало данных, простой read-heavy сценарий, важна скорость реализации.  
JSON-поля сократили сложность схемы и ускорили MVP. При росте проекта естественный следующий шаг — нормализация в `places` и `tips` с внешними ключами на `countries`.

---

#### Вопрос 2
**Как обеспечивается совместимость схемы БД между версиями без миграций уровня Flyway/Knex?**

**Ответ:**  
Сейчас используется lightweight-подход: при старте выполняется проверка колонок через `PRAGMA table_info`, и при необходимости — `ALTER TABLE ADD COLUMN`.  
Для production это заменяется полноценной системой versioned-миграций с rollback, но для учебного объема такой механизм достаточен.

---

#### Вопрос 3
**Почему выбран простой монолит (Express + статические страницы), а не SPA/SSR-фреймворк?**

**Ответ:**  
Цель проекта — минимальный backend, понятная архитектура и быстрый деплой.  
Монолит на Express закрывает эти требования: одна кодовая база, простая отладка, минимум инфраструктуры.  
Для текущего объема это эффективнее, чем добавлять избыточную сложность фреймворка.

---

#### Вопрос 4
**Как решена безопасность, если нет авторизации и расширенной серверной валидации?**

**Ответ:**  
Реализованы базовые практики: параметризованные SQL-запросы, базовая валидация обязательных полей, клиентское экранирование HTML.  
Проект не позиционируется как production-ready security-wise — это осознанные границы учебного scope.  
План развития: `helmet`, rate limiting, schema validation (`zod/joi`), auth и role-based доступ.

---

#### Вопрос 5
**Почему используется SQLite и открытие/закрытие соединения на запрос, а не PostgreSQL + pool?**

**Ответ:**  
SQLite выбран как zero-config база для локального и учебного сценария: один файл, быстрый запуск, минимальная операционная нагрузка.  
На текущей нагрузке такой подход рационален. При масштабировании архитектура эволюционирует в PostgreSQL + pooling.

### 12.2 Как профессионально защитить выбор технологий

Удобный шаблон ответа: **критерии выбора → контекст проекта → trade-offs**.

#### Node.js + Express
- **Критерий:** скорость разработки, простота API.
- **Контекст:** небольшой учебный backend.
- **Trade-off:** меньше архитектурной строгости, чем у Nest, но выше скорость реализации.

#### SQLite
- **Критерий:** минимум инфраструктуры.
- **Контекст:** малый объем данных и локальный запуск.
- **Trade-off:** ограниченная масштабируемость записи, но этого достаточно для текущей задачи.

#### Vanilla JS frontend
- **Критерий:** прозрачность и низкий порог сложности.
- **Контекст:** несколько страниц, несложное состояние.
- **Trade-off:** при росте UI целесообразен переход на компонентный framework.

Ключевая формулировка на защите:  
**«Стек выбран не “потому что проще”, а потому что он оптимален для масштаба и целей проекта; ограничения осознаны, путь эволюции спроектирован».**

### 12.3 Слабые места (техдолг) и как на них грамотно отвечать

#### 1) JSON в колонках `what_to_see`, `tips`
- **Риск:** сложнее делать аналитические SQL-запросы.
- **Позиция на защите:** осознанный компромисс ради скорости MVP; нормализация запланирована.

#### 2) Нет полноценной системы миграций
- **Риск:** сложнее сопровождать схему при росте.
- **Позиция на защите:** для учебного объема применен lightweight-подход; в production — versioned migrations.

#### 3) Ограниченная серверная валидация
- **Риск:** возможны неконсистентные данные.
- **Позиция на защите:** базовая валидация присутствует, строгая schema-validation — следующий этап.

#### 4) Нет auth/roles
- **Риск:** mutation endpoint открыт.
- **Позиция на защите:** в рамках проекта это допустимо; при расширении — auth + RBAC + закрытие write-эндпоинтов.

#### 5) Большой `server.js` (смешение слоев)
- **Риск:** рост сложности поддержки.
- **Позиция на защите:** допустимо для малого монолита; запланирован рефакторинг на слои `routes/services/repositories`.
